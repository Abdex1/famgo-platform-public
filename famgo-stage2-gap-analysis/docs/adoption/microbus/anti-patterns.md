# Microbus (microbus-io/fabric) — Anti-Patterns

Source: Stage 1 event-driven architecture notes. These are limitations Microbus's **own documentation** admits to — not external criticism — flagged so FamGo doesn't inherit them while adapting the patterns in `patterns.md`.

## Confirmed, Documented Limitations

1. **Circuit breaker with no auto-give-up.** Microbus's docs state directly: "the Foreman probes a tripped breaker indefinitely, so a task whose endpoint is permanently dead will park forever." Bounding this is left entirely to the workflow author via flow lifetime or an explicit timeout — i.e., it is opt-in safety, not a default. **FamGo implication**: when adapting Pattern 4 (typed workflow transition graph) and Pattern 6 in `patterns.md`, any circuit-breaker/parking mechanism FamGo builds into `platform/saga/` or `platform/resilience/circuitbreaker/` should default to a bounded probe lifetime, not require every workflow author to remember to add one.
2. **Parked backlog is not a real dead-letter queue.** Two material gaps versus a conventional DLQ: (a) parked work isn't redirected to a separate, inspectable channel for manual triage — it stays attached to its original workflow step, making bulk triage harder; (b) there is **no automatic expiry or poison-message threshold**. Microbus's design notes treat this as a deliberate scope boundary, not an oversight — meaning it is a known, accepted gap in the framework, not something to assume gets fixed in a newer version without checking.
3. **Orchestration-only model, no choreography option documented.** A single Foreman owns all graph execution, fan-out/fan-in, and state merging. This is a valid choice, but it does mean Microbus offers no native pattern for the alternative (services reacting to each other's events with no central coordinator) — if FamGo's ride lifecycle has any stages better suited to choreography (e.g., independent fraud/safety checks that don't need to block the main flow), Microbus's documentation provides no template for that and FamGo would need to design it independently.
4. **In-process short-circuit transport is publisher-attested, not broker-verified.** When two connectors share a process, NATS is bypassed entirely via a process-global trie — functionally equivalent addressing, but without the broker's verification. This is a stated trust trade-off (faster, less verified) that would need its own equivalent decision if FamGo ever builds an analogous in-process fast path for same-host service calls.

## Note on Domain Mismatch (not a flaw, but a calibration point)

Microbus is built for **agentic/LLM workflows**, not ride-hailing. Patterns like the auto-published OpenAPI-as-LLM-tool-surface and locality-aware routing for multi-region agent deployments are solving problems FamGo does not currently have evidence of needing (see migration-matrix.md REJECT rows for these two items specifically). Pulling them in "because they're well-documented" rather than because FamGo has the underlying need would itself be an anti-pattern — building speculative infrastructure ahead of a real requirement.
