Derived from `docs/adoption/gap-analysis.md`. Each row is a single component-level decision. `ADOPT` = bring in close to as-is; `ADAPT` = bring in the pattern/idea, re-implement for FamGo's stack (Go/Kafka/gRPC, not Node/NATS); `REJECT` = do not bring in, with reason.

## Notes on Using This Matrix

1. **ADAPT is the dominant decision**, not ADOPT — almost nothing here should be copy-pasted. Both source repos differ from FamGo's stack (richxcame: different language conventions and an HTTP-only mesh; Microbus: different transport (NATS) and a different domain (agentic workflows, not ride-hailing)).
2. Several target locations recur across multiple rows (`platform/saga/`, `platform/outbox/`, `packages/event-bus/contracts/`, `platform/resilience/ratelimit/`) because they are currently **empty directories** in FamGo — these are the highest-leverage rows in the matrix since they convert "scaffolded" into "populated."
3. Every REJECT row states a reason; none are rejected merely because they're unfamiliar — each ties back to a specific finding in `gap-analysis.md`.
