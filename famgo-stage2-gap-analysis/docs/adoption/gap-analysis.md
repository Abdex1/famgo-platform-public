# FamGo Gap Analysis — Stage 2

**Inputs compared:**
- FamGo repository structure — `project-structure-latest.txt` (flat path listing, no file contents; ~3,700 paths under `C:\dev\FamGo-consolidated`)
- `richxcame/ride-hailing` — Service, API, Event, Database, Realtime, Infrastructure inventories (Stage 1)
- `Autofleet/locomotion` — Ride lifecycle workflow analysis (Stage 1)
- `microbus-io/fabric` — Event-driven architecture notes (Stage 1)

**Method note (read before the tables):** The FamGo side of this analysis is built from a **path listing only** — no file contents, no `go.mod` dependency graphs, no running code. A path such as `platform/saga/` proves a directory was scaffolded; it does not by itself prove a saga engine exists or works. Where a directory contains **zero child files**, that is stated explicitly and treated as "scaffolded, not implemented." Where a directory contains `.go` files, that is treated as "code present" — not as "code correct, tested, or wired into a running service," since contents were not read. Every row below is traceable to a specific path or to a specific sentence in a Stage 1 document; nothing below is inferred from general ride-hailing-platform knowledge.

## 0. Snapshot: What FamGo Already Has That the Source Repos Mostly Don't

This matters for calibrating the rest of the document. Unlike `richxcame` (HTTP-only, NATS disabled by default) and `Autofleet/locomotion` (thin rider client, no backend in-repo), FamGo's structure already shows:

- A **dedicated Kafka SDK package** (`packages/kafka-sdk/`) with producer, consumer, DLQ handler, retry handler, schema, versioning, and outbox sub-packages — richxcame has no equivalent; it provisions NATS but ships it disabled.
- A **dedicated event-bus contracts package** (`packages/event-bus/`) with per-domain contract folders (`auth`, `dispatch`, `driver`, `notification`, `payment`, `ride`, `safety`), an envelope type, governance/naming/ownership/retention files, a topic registry, and a versioning strategy file — this is structurally ahead of both richxcame (no contracts at all) and Microbus (contracts exist but as generated client stubs from `api` sub-packages, a different mechanism).
- A **DDD-layered service template** (`internal/domain/`, `internal/application/`, `internal/infrastructure/`, `internal/interfaces/{grpc,rest}/`) applied consistently to `ride-service`, `dispatch-service`, `auth-service`, `user-service` — more structured than richxcame's flatter `internal/<service>` layout.
- `ride-service` specifically has `application/saga.go`, `application/events.go`, and `application/event_subscribers.go` — i.e., FamGo's most mature service already has saga and event-subscriber files where richxcame's Rides service has neither (richxcame's ride lifecycle is plain synchronous REST).

The gaps documented below are therefore mostly about **breadth and depth of what's already started**, not a green-field absence of architecture.

**Headline event-architecture finding:** FamGo's event layer is **better-designed on paper than richxcame's is in practice**, but a meaningful fraction of FamGo's own event directories (`platform/saga/`, `platform/outbox/`, `platform/event-bus/`, `platform/event-governance/`, `platform/cdc/`, several `packages/kafka-sdk/internal/*` subfolders, several `packages/event-bus/contracts/*` domain folders, all of `standards/events/{contracts,dlq,retry-policies,schemas,topics}/`) are **directories with no files in them** — i.e., named intent, not delivered capability. This is the single most important fact for the Migration Matrix and Extraction Packages: FamGo does not primarily need to *import* event-architecture ideas from richxcame (which has fewer of them) — it needs to **finish populating its own scaffolding**, and Microbus is the more useful reference for *how* to do that (bounded retries, structured topic addressing, typed workflow transitions) than richxcame is.
