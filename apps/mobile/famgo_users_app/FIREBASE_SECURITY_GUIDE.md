# Production-Grade Firebase Realtime Database Security Rules

## 📋 Overview

This document provides enterprise-level security rules for a ride-sharing application using Firebase Realtime Database. The rules implement:

- **Authentication-based access control**
- **Field-level validation**
- **Data structure enforcement**
- **Rate limiting & quota protection**
- **Admin controls & block lists**
- **Comprehensive audit trails**

---

## 🔒 Security Principles

### 1. **Principle of Least Privilege**
- Users can only access their own data
- Drivers can only update their own availability
- Only admins can modify sensitive fields (blockStatus, verification)

### 2. **Data Validation at Database Level**
- Type checking (string, number, boolean)
- Format validation (regex for email, phone, coordinates)
- Range validation (latitude -90 to 90, longitude -180 to 180)
- Required fields enforcement
- Prevent injection attacks

### 3. **Immutability for Critical Fields**
- User ID (created once, never modified)
- Email (created once, prevents account hijacking)
- License number (created once for driver verification)
- Vehicle number (created once for compliance)
- Timestamps (server-set, not user-controllable)

### 4. **Admin Separation**
- Separate `admins` collection for authorization
- Only admins can block/unblock users
- Only admins can verify drivers
- Audit trail for admin actions

## 🛡️ Detailed Security Rules

### 1. Users Collection

**Purpose**: Store passenger and driver profile information

**Access Rules**:
```
- Read: User can only read their own profile
- Write: User can only modify their own data (except blockStatus)
- blockStatus: Only admins can modify
- Email: Immutable after creation
```

**Validation Examples**:
```json
{
  "name": "John Doe",          // Required, 3-100 chars, letters/spaces only
  "email": "john@example.com", // Required, valid email format, immutable
  "phone": "+12345678900",     // Required, E.164 format
  "blockStatus": "no",         // 'yes' or 'no', admin-only
  "userType": "passenger",     // 'passenger' or 'driver', immutable
  "createdAt": 1694000000000,  // Server timestamp
  "updatedAt": 1694000000000   // Server timestamp
}
```

### 2. Drivers Collection

**Purpose**: Store driver-specific information (license, vehicle, ratings)

**Access Rules**:
```
- Read: Driver can read own data; public can read if isAvailable=true
- Write: Driver can update own data; admins can update ratings
- License: Immutable after creation
- Vehicle: Immutable after creation
- isAvailable: Driver can toggle; blocked drivers cannot
```

**Validation Examples**:
```json
{
  "licenseNumber": "DL123456",     // Required, immutable, 5-20 chars
  "licenseExpiry": "2025-12-31",   // Must be future date
  "vehicleNumber": "MH01AB1234",   // Required, immutable, uppercase
  "vehicleType": "car",             // 'car', 'auto', or 'bike'
  "isAvailable": true,              // Boolean, updated in real-time
  "latitude": 28.7041,              // Current position
  "longitude": 77.1025,             // Current position
  "rating": 4.8,                    // Admin-set, 0-5 range
  "totalTrips": 150,                // Admin-set counter
  "bankDetails": {                  // Optional, driver payout info
    "accountNumber": "1234567890",
    "bankName": "Example Bank"
  }
}
```

### 3. Online Drivers Collection (GeoFire)

**Purpose**: Real-time location tracking for driver-passenger matching

**Access Rules**:
```
- Read: Public (for passenger matching)
- Write: Driver can update own location
- Only drivers who are not blocked can write
```

**Structure**:
```json
{
  "driverId1": {
    "g": "u4...",  // GeoHash (generated by GeoFire)
    "l": [28.7041, 77.1025]  // [latitude, longitude]
  }
}
```

### 4. Trip Request Collection

**Purpose**: Manage ride requests, matching, and trip lifecycle

**Access Rules**:
```
- Read: Only involved parties (user, driver) can read
- Write: User creates trip; driver/user update status
- Immutable fields: userID, pickUp/dropOff locations
```

**Status Flow**:
```
new → accepted → arrived → ontrip → ended
      ↓
    cancelled
```

**Validation Examples**:
```json
{
  "tripID": "trip_123",
  "userID": "user_456",           // Immutable, set by user on creation
  "userName": "John Doe",
  "userPhone": "+12345678900",
  "pickUpLatLng": {               // Immutable
    "latitude": 28.7041,
    "longitude": 77.1025
  },
  "dropOffLatLng": {              // Immutable
    "latitude": 28.5355,
    "longitude": 77.3910
  },
  "vehicleType": "car",
  "fareAmount": 250.50,           // Must be 0-100000
  "bidAmount": 200.00,            // Optional, passenger can bid lower
  "status": "ontrip",             // Enum validation
  "driverId": "driver_789",
  "paymentMethod": "cash",        // 'cash', 'card', 'wallet'
  "createdAt": 1694000000000
}
```

### 5. Ratings Collection

**Purpose**: Store trip reviews and ratings

**Access Rules**:
```
- Read: Public
- Write: Only the user who took the trip can rate
- Immutable: userId, tripId (can only rate once per trip)
```

**Validation**:
```json
{
  "userId": "user_456",           // Immutable, auth.uid
  "driverId": "driver_789",
  "tripId": "trip_123",           // Immutable, each user rates once
  "rating": 4,                    // 1-5 only
  "review": "Great service!",     // Optional, max 500 chars
  "createdAt": 1694000000000
}
```

### 6. Payments Collection

**Purpose**: Transaction history and payment tracking

**Access Rules**:
```
- Read: User can read their own; driver can read their own; admins see all
- Write: Auto-created by backend; admins can update status
- Audit trail: All payment records are immutable
```

## 🚨 Security Features

### 1. **Block List Protection**
- Users in `_metadata.blockList` cannot:
  - Create trip requests
  - Update driver availability
  - Write to online drivers location
  - Modify any data

### 2. **Field Validation**
- **Email**: RFC 5322 compliant regex
- **Phone**: E.164 international format
- **Coordinates**: Valid latitude (-90 to 90) and longitude (-180 to 180)
- **Enum fields**: Only predefined values allowed
- **Timestamps**: Must be ≤ current time (no future dates)

### 3. **Immutable Fields**
Critical fields that cannot be changed after creation:
- User ID
- Email address
- Driver license number
- Vehicle registration number
- User type (passenger/driver)

### 4. **Admin-Only Fields**
Fields only admins can modify:
- `blockStatus` - Block/unblock users
- `verificationStatus` - Driver verification
- `rating` - Official driver rating
- `totalTrips` - Trip counter

### 5. **Data Integrity**
- Prevent orphaned records
- Enforce required fields
- Validate relationships
- Deny unknown fields (`$other: .validate = false`)

## 📝 Implementation Checklist

### Step 1: Deploy Rules
1. Go to Firebase Console
2. Select your project
3. Navigate to Realtime Database → Rules
4. Copy the entire `firebase_rules_production.json` content
5. Paste into the rules editor
6. Click "Publish"

### Step 2: Create Admin Users
```javascript
// In Firebase Console, manually add to /admins collection:
firebase.database().ref('admins/admin_uid_here').set(true);
```

### Step 3: Set Initial Metadata
```javascript
firebase.database().ref('_metadata').set({
  blockList: {},
  stats: {
    totalUsers: 0,
    totalDrivers: 0,
    totalTrips: 0
  }
});
```

### Step 4: Update Client Code
Ensure your Flutter app:
- Catches exceptions from validation failures
- Validates data before writing
- Handles auth errors properly
- Implements proper error messages

### Step 5: Monitoring
- Monitor Firebase logs for rule violations
- Track block list changes
- Monitor failed writes
- Track admin actions

## ⚠️ Important Notes

### What These Rules Protect Against:

✅ **Unauthorized access** - Users can't read/write other users' data
✅ **Data injection** - Invalid formats rejected at DB level
✅ **Account takeover** - Email/ID immutable after creation
✅ **Fraud** - Transaction amounts limited, status validation
✅ **Mass modification** - Block list prevents blocked users from acting
✅ **Data corruption** - Required fields enforced, structure validated

### What These Rules DON'T Protect Against:

⚠️ **Rate limiting** - No per-request rate limit (implement in backend)
⚠️ **DDoS attacks** - Implement at infrastructure level
⚠️ **Stolen auth tokens** - Implement token rotation in backend
⚠️ **Business logic** - Implement matching algorithm in backend
⚠️ **Payment verification** - Verify with payment gateway

### Test Read Access
```javascript

db.ref('users/user1').once('value') → ✅ Success

// User 1 cannot read User 2's data
db.ref('users/user2').once('value') → ❌ Denied
```

### Test Write Validation
```javascript

// Invalid email format
db.ref('users/user1').update({email: "invalid"}) → ❌ Validation error

// Invalid phone format
db.ref('users/user1').update({phone: "123"}) → ❌ Validation error

// Valid coordinates
db.ref('drivers/driver1').update({latitude: 28.7041, longitude: 77.1025}) → ✅

// Invalid latitude (> 90)
db.ref('drivers/driver1').update({latitude: 100}) → ❌ Validation error
```

## 📞 Troubleshooting

### Common Issues

| Issue | Cause | Solution |
|-------|-------|----------|
| Permission denied | Not authenticated | Ensure user is logged in via Firebase Auth |
| Validation error | Invalid data format | Check field format (email, phone, coordinates) |
| Field write blocked | Field is immutable | Don't try to modify immutable fields after creation |
| Block list error | User is blocked | Remove from `/admins/blockList` |
| No onlineDrivers visible | Driver availability not updated | Ensure driver sets `isAvailable: true` |

## 🎯 Best Practices

### 1. **Client-Side Validation**
```dart
// Before writing to DB, validate on client:
if (!isValidEmail(email)) {
  showError("Invalid email format");
  return;
}

if (!isValidPhone(phone)) {
  showError("Invalid phone format");
  return;
}
```

### 2. **Error Handling**
```dart
try {
  await db.ref('users/$uid').set(userData);
} on FirebaseException catch (e) {
  if (e.code == 'PERMISSION_DENIED') {
    showError("You don't have permission");
  } else if (e.code == 'VALIDATION_FAILED') {
    showError("Invalid data format");
  }
}
```

### 3. **Immutable ID Pattern**
```dart
// When creating a user, set ID to auth.uid
Map<String, dynamic> userData = {
  'id': auth.uid,  // ← Never allow user to change this
  'name': userName,
  'email': userEmail,
  'createdAt': DateTime.now().millisecondsSinceEpoch,
};
```

### 4. **Secure BlockList Check**
```dart
// Backend should check block status on every request:
bool isUserBlocked = await checkBlockStatus(userId);
if (isUserBlocked) {
  return error("User is blocked");
}
```

## 📈 Scaling Considerations

### For 1M+ Users:
1. **Implement Realtime Database backups**
2. **Archive old trips to Cloud Storage**
3. **Use Cloud Functions for heavy computation**
4. **Implement caching layer**
5. **Separate read/write paths**

### Firestore Alternative:
Consider migrating to Firestore for:
- Stronger consistency guarantees
- Better scalability
- Easier data aggregation queries
- Offline support
